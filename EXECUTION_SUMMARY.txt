### 🎯 KEY FEATURES IMPLEMENTED

#### A. Token-Aware Input Truncation
```
Problem:  Input data too large (~8,000-12,000 tokens) for 8000 TPM limit
Solution: Aggressive truncation with prioritization

Budget Allocation (~4,000 tokens):
├─ System message + schema: ~8,000 chars
├─ Vulnerable snippet: KEEP INTACT (critical)
├─ Raw text: Truncate to 60% of remaining (~3,000 chars)
├─ Surrounding context: Truncate to 40% (~2,000 chars)
└─ Reserve: ~4,000 tokens for model output

Result:   2-3x more records processable within 8000 TPM
```

#### B. Exponential Backoff for Rate Limiting
```
All keys hit 429? → Wait 60s  → Retry
All keys hit 429? → Wait 120s → Retry
All keys hit 429? → Wait 300s → Retry
Max retries? → Fail record, move to next
```

#### C. Multi-Key Rotation (8 keys from .env)
```
Your .env contains: 8 Groq API keys (gsk_*)
Rotation strategy: Round-robin through all 8 keys
Failure handling: Per-key cooldown tracking
Recovery: Automatic when key cooldown expires
```

### ⚙️ TECHNICAL DETAILS

**Groq Configuration**:
- API Base: `https://api.groq.com/openai/v1`
- Model: `deepseek-r1-distill-llama-70b`
- Endpoint: `/chat/completions`
- TPM Limit: 8,000 tokens per minute
- Input Budget: ~4,000 tokens
- Output Budget: ~4,000 tokens (reserved)

**Token Truncation**:
- Algorithm: Keep snippet intact, truncate context by ratio
- Raw text truncation: ~3,000 chars max (60% of context budget)
- Context truncation: ~2,000 chars max (40% of context budget)
- Marker: `[truncated]` appended to truncated fields
- Logging: `[TRUNCATE]` prefix shows before/after sizes

**Rate Limiting**:
- Strategy: Per-key cooldown with exponential backoff
- Backoff sequence: 60s → 120s → 300s (capped)
- Max retries: 3 full rotations through all 8 keys
- User control: Can Ctrl+C to interrupt sleep
- Recovery: Automatic retry after backoff duration
